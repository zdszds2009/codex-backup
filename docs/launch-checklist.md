# Launch Checklist

Use this before making the repository public and before applying to `Codex for Open Source`.

## Must-have

- Public GitHub repository
- Public maintainer profile
- `README.md`, `LICENSE`, `SECURITY.md`, `CONTRIBUTING.md`
- At least one test command that passes on a clean local environment
- One screenshot, GIF, or short video showing package creation and restore

## Strongly recommended

- First tagged release, for example `v0.1.0`
- A short release note describing the workflow and limitations
- At least one sample issue opened by the maintainer to show project direction
- A short note on privacy and over-collection risk
- Clear explanation that project-file inclusion is explicit, not automatic for thread-only exports

## Evidence that improves application odds

- Stars from users outside your own accounts
- One or more issue reports or feature requests from real users
- A public usage example, blog post, thread, or video
- A maintainer story explaining why manual `.codex` recovery was painful before this tool
- Evidence that the project solves a repeated problem, not a one-off personal backup

## Submission timing

Apply after the repository is public and the first release is live. Applying before that is possible, but your odds are materially lower because reviewers cannot see public adoption signals.
